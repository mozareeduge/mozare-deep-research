# Methodology adaptation

The source methodology is kept as human-facing documentation outside the operational context. MROS transfers its protections without asking a model to reproduce its vocabulary.

## Transfer pattern

| Human concern | Operational implementation |
|---|---|
| Materials arrive through conditions | `SourceRecord` stores access, version, transformation, rights and provenance |
| A connection should precede a claim | `CandidateLink` has support, challenges, status and allowed uses |
| Evidence controls language | `EvidenceCard`, `ClaimRecord`, export status and release audit |
| Generated text must not become evidence | model validators and audit blockers |
| Failure and omission remain available | rejected/deferred ledger, reopening condition and event log |
| Formation should be reconstructable | hash-chained events, handoff receipts and versioned records |
| Analogy should do work | `DesignDecision` requires operation, consequence, test and failure risk |
| Making can test an uncertainty | `prototype-test` run profile and design records |
| Genre changes selection | outline and release skills require audience, function and omission records |

## Anti-misdirection controls

- No mandatory generation of new connections.
- No global numeric confidence scores.
- No model self-promotion from lead to evidence or evidence to claim.
- No use of model memory as formal evidence.
- No whole-methodology prompt in every session.
- No need to use every source, passage or connection.
- Operational workers return typed records and short rationales rather than methodological essays.
